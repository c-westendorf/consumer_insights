Phase 01 · Define · Features
F1
Scope Generation
Intent classification agent invisibly fills a CGF-governed scope template through multi-turn conversation. Always multi-pass. The template is invisible to the stakeholder — they ask questions, the agent extracts structure. Stakeholder confirms at the end.
Inputs
  • Stakeholder business question (natural language)
  • CGF scope template
  • Semantic layer (governed definitions)
  • Ontology layer
Outputs
  • Validated scope document
  • All CGF variables populated
  • Sampling logic locked
  • Stakeholder confirmation received
✓ Pass gateAll template variables filled, resolvable in data lake, stakeholder confirms
✗ Fail gateStakeholder cannot specify enough fidelity → report out, request refinement

F1b
Plan Generation
Rules-based system matches scope to analysis types, selects playbooks, sequences them. Two outputs: a simple numbered plan for stakeholders and a dependency work tree for execution.
Inputs
  • Validated scope document
  • Analysis type library (F6)
  • Playbook library (~10 playbooks)
  • Rules-based mapping: scope variables → analysis types → playbooks
Outputs
  • Numbered plan artifact (stakeholder-facing: 1…N)
  • Work tree (execution: sequential + parallel, dependencies)
  • Input contract for Phase 2
✓ Pass gateEvery step has an assigned playbook; every input traces to scope variables or prior outputs; no orphan steps
✗ Fail gateScope matches no known analysis type → flag for manual planning
Builder spec — work tree logic & rules-based selection›
Work Tree vs Numbered Plan

The stakeholder sees 1, 2, 3, 4, 5. The work tree underneath shows parallelism, hard dependencies, where output of one playbook feeds input of the next, and where failure cascades. Changing sampling logic mid-analysis is rewriting a function argument — if scope variables change after the plan is generated, the entire work tree must be re-evaluated. This is the lesson from analyses 1 and 2.

Rules-Based Selection (PoC)

The four analyses are the training material. For each, reverse-engineer: what was the question type, what playbooks ran, in what order, where did they loop, what grading criteria determined advancement? That pattern becomes a rule. The PoC supports four known analysis types. Generalization to a learned algorithm is post-PoC.

Inputs

  • Semantic layer
  • Ontology
  • Prior decision logs from past analyses
Outputs

  • Data source manifest (table, field, freshness, access)

✓ Pass gateEvery variable maps 1:1 to a governed definition; all data sources accessible and within freshness requirements
✗ Fail gateVariable undefined in semantic layer → flag for governance; stale or inaccessible data → report out

Phase 02 · Analyze · Features
F3
Data Agent
Retrieves, validates, and prepares data using the scope and data source manifest. First hard stop: can the data actually answer the business question? Not all questions are answerable with available data — this is where that becomes explicit.
Inputs
  • Scope document (segment definitions, time windows, sampling logic)
  • Data source manifest from F2
  • Playbook expected schema
Outputs
  • Analysis-ready dataset
  • Data quality report
  • Schema conforming to playbook input expectations
  • Validation pass/fail report
✓ Pass gateData supports outcome measures in scope; volume sufficient for statistical significance; schema matches playbook expectations
✗ Hard stop failData cannot answer the business question → "we cannot conclude on this with available data" — analysis does not proceed
Builder spec — three-stage validation›
Three-Stage Validation
S1Data retrieval — Query data lake using resolved mappings, apply sampling logic. Gate: row counts in range, time windows correct, segment definitions match.
S2Answerability / first hard gate — Sufficient volume for stat significance? Key outcome variables populated? Fail → hard stop, report out. This is a process terminator.
S3Data preparation — Clean, transform, join, structure. Apply standard transformations defined in playbooks. Output schema must match what downstream playbooks declare as required inputs.
F4
Playbook Execution Framework
Executes playbooks per the work tree — sequentially or in parallel. Each playbook is a while-loop: explore a hypothesis, evaluate against grading criteria, pass forward or loop back, hit a max iteration ceiling, fail out with a report.
Playbook anatomy
  • Name and type — EDA playbook, feature engineering, interval analysis, etc.
  • Required inputs — typed; trace to scope variables or prior playbook outputs; untraceable inputs = broken work tree
  • Sequential steps with defined completion states
  • Grading criteria per step — what "good enough to proceed" looks like; not vague
  • Loop-back logic — if criteria fail, what does the analyst re-explore and what changes?
  • Max iteration count — ceiling before declaring approach is not working
  • Failure report template — what was tried, why it didn't work, what that means for the question
  • Defined outputs — typed, format downstream playbooks expect
✓ Pass gateStep output meets grading criteria → feeds next playbook
✗ Fail gateMax iterations exceeded → generate failure report → escalate or report out
Builder spec — explore-exploit while-loop (6 steps)›
The While-Loop
  • Enter step with hypothesis — what are we exploring, what do we expect, captured in reasoning scaffold in real time
  • Execute step against the data — EDA, feature engineering, statistical test, cohort comparison
  • Evaluate against grading criteria — pass → capture result and move forward; fail → log why and loop back
  • Loop back if fail — adjust approach, re-execute; each attempt logged in decision log
  • Max iterations reached — fail out with report: what was tried, why nothing worked, what this means for the question. A legitimate analytical conclusion.
  • Pass → produce outputs, feed next playbook in work tree
F5
Reasoning Scaffolding
Captures the explore-exploit loop in real time. Every hypothesis, method, result, trade-off decision, and failed approach is logged. The decision log generated here is what makes the agent smarter over time.
Captured at each iteration
  • Hypothesis — what the analyst was exploring and why
  • Method applied — technique, transformation, or statistical test
  • Result observed — what the data showed
  • Grading evaluation — did it pass or fail criteria and why?
  • Decision made — continue, loop back, pivot, or fail out
  • Trade-off reasoning — why X over Y; this is the core of the decision log
Builder spec — decision log material & three artifact relationship›
Decision Log — What Gets Tagged
  • Feature engineering choices — why engagement-to-value conversion was approached this way vs. alternatives
  • Causal linkage reasoning — why this causal framing vs. correlational, what confounds were controlled
  • Sampling decisions — why this cohort definition, what edge cases were excluded
  • Attribution methodology — why this psychometric framing for need-state classification
Three Core Artifacts
ScopeBoundary of the work — what are we answering
PlaybooksGrading criteria and structure of the while-loop
Decision logReasoning that fills the space between playbook steps — why X over Y at each choice point

Together these are the intellectual scaffolding. Runbook and conclusion are execution outputs.

F6
Analysis Type Library
Catalog of every analysis the team performs, each mapped to a playbook sequence with defined inputs and grading criteria. Powers the rules-based plan generator in Phase 1. Built through the four analyses.
Builder spec — all 8 analysis types with playbook sequences›
8 Known Analysis Types
A1
Cohort behavior analysis
Observe a defined cohort over a time window, identify behavioral patterns
data retrieval → EDA → behavioral segmentation → pattern identification → statistical validation → conclusion
A2
Stickiness / interval analysis
Measure engagement frequency and consistency over time
data retrieval → EDA → time-series feature engineering → interval pattern analysis → stickiness scoring → statistical validation → conclusion
A3
Engagement decay analysis
Customers moving active → disengaged (BNO: 13-week non-engagement threshold, scored at week level)
data retrieval → EDA → engagement trajectory mapping → decay point identification → leading indicator extraction → conclusion
A4
Reactivation behavior analysis
Customers who re-engaged after disengagement but before churn (52-week threshold)
data retrieval → EDA → reactivation trigger identification → post-reactivation behavior profiling → value recovery assessment → conclusion
A5
Attribution / maturity modeling
Psychometric mapping of outcome variables to underlying behavioral need states
data retrieval → EDA → feature engineering → outcome variable clustering → attribution label assignment → validation → conclusion
A6
Value decomposition analysis
Quantify where the org is growing or shrinking in customer counts, volume, and associated value
data retrieval → EDA → decomposition by segment dimensions → value change calculation → growth/shrink attribution → conclusion
A7
Leading indicator development
Identify behavioral signals predicting future outcomes: value change, churn risk, engagement trajectory
data retrieval → EDA → feature engineering → predictive feature selection → validation against outcomes → indicator scoring → conclusion
A8
Causal linkage analysis
Establish or test causal relationships between behaviors and outcomes
data retrieval → EDA → confound identification → causal method selection → analysis execution → robustness checks → conclusion

Target ~10 reusable playbooks. EDA playbook is shared across all 8 types. Team's job in analyses 3 and 4 is to exhaust this inventory.

Inputs

Outputs

  • Visual package (charts, tables, segment comparisons)
  • Full deliverable with lineage + traceability attached
  • Lineage: insight → playbook → data source → scope variable
✓ Pass gateEvery scoped business question addressed; hypotheses confirmed, rejected, or marked inconclusive with reasoning
✗ Fail gateNarrative cannot coherently connect results to scoped questions → flag for analyst, potentially loop back to Phase 2

F8
Knowledge Layer Persistence
Curated learnings committed back to the semantic layer and ontology after each analysis. Closes the loop from Phase 3 back to Phase 1. Manually curated — an analyst reviews before committing. Not everything generalizes.
What gets committed
  • Playbook refinements — versioned updates to playbook library
  • Decision log entries — tagged to analysis type, queryable by future agents; especially feature engineering, causal reasoning, attribution methodology choices
  • Conclusion patterns — indexed by question type; what question types led to what conclusion types
  • Failure cases — where analyses couldn't conclude, what data was missing — equally valuable as successes
Builder spec — curation principle›
Curation Principle

Not everything generalizes. An insight specific to medium-engaged customers in Q3 is not automatically useful for a new customer cohort analysis. The curation step separates generalizable institutional memory from run-specific detail. This manual review step must be preserved even after automation — it is the quality gate for the knowledge layer.

The gap: systems thinking at the function level

The gap the team is experiencing is not about understanding the phases at a high level — they understand those. The gap is systems thinking at the function level. Every step in their manual process is a function with typed inputs, typed outputs, and dependencies.

When they redefined sampling between analyses, they were rewriting the same function instead of parameterizing it once. The visual they need to build is not a narrative flow — it is a dependency graph of functions: scope generates variables that feed knowledge layer resolution that generates a manifest that feeds data retrieval that produces a schema that feeds playbook step 1 whose output feeds playbook step 2, and so on.

Break any link in that chain and the analysis breaks. That is the moment of understanding they need.
