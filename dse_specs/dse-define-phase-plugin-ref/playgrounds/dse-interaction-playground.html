Follow-up handling — the thread already exists

Claude Code and Cowork own the conversation. The plugin shouldn't reinvent session memory — it should let the skills branch inside the thread that's already there.

The mistake would be treating /scope as the only door in. In-thread, a follow-up like "now decompose that by tenure bucket" is not a new scope — it's a refinement of the scope that's already sitting in context two turns up. The scope-agent and plan-agent should recognize that and branch, not cold-restart.

What the host gives us free

Full conversation history. The prior scope_artifact.json is on disk and the narrative is in the thread. No session store needed — Claude can read both.

What the plugin must do

Design skills that branch naturally: detect existing scope in context, offer refine / extend / fork paths, and skip the cold clarifier tax when the answer is already one artifact away.

Skill affordances to add

scope-refinement (narrow an existing scope without re-asking intent), plan-extension (append a branch to a live plan), result-decomposition (split a delivered metric along a new dimension). All read the most recent artifact from disk; none require a session index.

What that changes in the flows above

Follow-ups cost 1–2 turns, not 5. The user says "decompose by tenure," scope-refinement reuses the locked scope, plan-extension appends one SQL pass, and results flow back. No new /scope, no re-clarification, no new decision log entry — just an amended one.
