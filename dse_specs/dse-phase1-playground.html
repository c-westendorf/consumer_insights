Phase 1: Define

What are we building and why?

We're building a system that turns a business question — asked in plain English — into a data science project plan that a principal data scientist would approve. No manual scoping. No back-and-forth in docs. Just a conversation that produces a real, executable plan.

🎯

The Problem

Data science projects start with vague business questions. Turning those into precise, well-planned work takes senior expertise and multiple rounds of clarification — and it's the step most likely to go wrong.

Misscoped projects waste everything downstream
💡

Our Approach

Two specialized agents work in sequence. The first has a conversation with you to understand exactly what you need. The second takes that understanding and designs the project plan — tasks, dependencies, and all.

Scope and planning are deliberately separated
✅

What Success Looks Like

A principal data scientist reviews the plan and says: "Yes, this is how I'd approach it." The plan is detailed enough that an execution system can pick it up and run it without asking clarifying questions.

Principal-quality plans from conversation
🧬

CGF-Native

The system speaks Customer Growth Framework. It knows lifecycle stages, customer segments, and the analytical patterns that matter for our business. It doesn't start from scratch — it starts from domain knowledge.

Domain-aware, not generic
📐

Phase 1 of 3

Scope Agent — the conversation partner

The Scope Agent's only job is to turn a vague business question into a precise, complete scope. It talks to the user, asks the right questions, and produces a structured document that says exactly what needs to be analyzed.

What It Does

Three things, nothing more

The Scope Agent classifies the intent (what kind of analysis is this?), extracts structured fields from the conversation (who, what, when, vs. what?), and validates that nothing critical is missing before locking the scope.

Classifies intent Extracts scope fields Validates completeness
What It Doesn't Do

Strict boundaries

It never decides how the work will be done. It never checks if data exists. It never selects tools or playbooks. Those are the Plan Agent's concerns. This separation means the scope stays pure — it captures the business need without being shaped by implementation constraints.

Conversation Design

One question at a time

The agent asks targeted questions to fill specific gaps. It doesn't dump a questionnaire. If you say "how are reactivated customers performing?", it knows the lifecycle stage (reactivated) and needs to clarify: performing on what metric? compared to whom? over what timeframe?

Toolkit — 4 specialized tools
🏷️ Intent Classifier

Maps question to CGF lifecycle stage and analytical pattern

📝 Scope Extractor

Parses population, metric, timeframe, comparison from conversation

✅ Completeness Validator

Checks all required fields are present for the analytical pattern

📋 Scope Formatter

Renders the final Scope Artifact for approval

Plan Agent — the project architect

The Plan Agent takes the locked scope and designs the project. It figures out what tasks are needed, which ones can run in parallel, what each task needs to finish, and how much exploration to budget. It never talks to the user — it works from the scope alone.

What It Produces

A plan with three views

The same plan serves three audiences. Stakeholders see a numbered list they can understand. The execution system sees a dependency graph it can run. And a trace view fills in during execution to create an audit trail.

Stakeholder Numbered steps in plain language — "First we'll profile the cohort, then compare spend patterns…"
Agent DAG of tasks with entry conditions, exit conditions, fan-out rules, and budget allocations
Trace Empty at plan time, fills in during execution — what happened, what was found, why it stopped
Key Concepts

Fan-out, while-loops, and budgets

Fan-out: Some tasks can run in parallel — analyzing four customer segments simultaneously instead of sequentially.

While-loops: Exploration tasks repeat until the agent reaches a conclusion or runs out of budget.

Budgets: Each task gets an exploration budget based on its type and importance. This prevents the system from exploring forever.

⚖️ Budget Allocator

Assigns exploration budgets and exit conditions per task

The contract between phases

Two artifacts are the formal output of Phase 1. They carry everything Phase 2 needs to begin execution — no ambiguity, no missing pieces, no verbal agreements. Click either artifact to see its structure.

  • 👥Population — exactly who we're analyzing

The Boundary Rule

Why the separation matters

The Scope Artifact is the only interface between the Scope Agent and the Plan Agent. They share no state, no conversation history, no side channels. This means: the scope can't be subtly altered during planning, the plan is always traceable to a specific approved scope, and if the plan is wrong, you know the problem is in planning — not in a scope that drifted.
